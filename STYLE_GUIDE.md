# MCP Server Style Guide

Based on the patterns observed and best practices, I recommend adopting the following consistent style guide across all your MCP servers:

| Element              | Convention                                                                                                                                    | Rationale / Examples                                                                                                                              |
| :------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------ |
| **CLI Commands**     | `verb-noun` in `kebab-case`. Use the shortest unambiguous verb (`ls`, `get`, `create`, `add`, `exec`, `search`).                              | `ls-repos`, `get-pr`, `create-comment`, `exec-command`                                                                                            |
| **CLI Options**      | `--kebab-case`. Be specific (e.g., `--workspace-slug`, not just `--slug`).                                                                    | `--project-key-or-id`, `--source-branch`                                                                                                          |
| **MCP Tool Names**   | `<namespace>_<verb>_<noun>` in `snake_case`. Use a concise namespace. Avoid noun repetition.                                                  | `machship_get_consignment`, `machship_create_quote`, `machship_search_locations`                                                                  |
| **MCP Arguments**    | `camelCase`. Suffix identifiers consistently (e.g., `Id`, `Key`, `Slug`). Avoid abbreviations unless universal.                               | `consignmentId`, `companyId`, `reference1`                                                                                                        |
| **Boolean Args**     | Use verb prefixes for clarity (`includeXxx`, `launchBrowser`). Avoid bare adjectives (`--https`).                                             | `includeExtendedData: boolean`, `launchBrowser: boolean`                                                                                          |
| **Array Args**       | Use plural names (`consignmentIds`, `labels`, `statuses`).                                                                                    | `consignmentIds: string[]`, `statuses: string[]`                                                                                                  |
| **Descriptions**     | **Start with an imperative verb.** Keep the first sentence concise (≤120 chars). Add 1-2 sentences detail. Mention pre-requisites/notes last. | `List available consignments. Filters by status, carrier, or date range. Returns formatted list including ID, reference, and status.`             |
| **Arg Descriptions** | Start lowercase, explain purpose clearly. Mention defaults or constraints.                                                                    | `numeric ID of the consignment to retrieve. Required.`                                                                                            |
| **ID/Key Naming**    | Use consistent suffixes like `Id`, `Key`, `Slug`, `KeyOrId` where appropriate.                                                                | `consignmentId`, `companyId`, `carrierId`                                                                                                         |

Adopting this guide will make the tools more predictable and easier for both humans and AI agents to understand and use correctly.
